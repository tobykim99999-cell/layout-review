# Agent 排版审核项目正规开发步骤

## 1. 明确目标与边界

本项目目标是替代毕业论文二次排版审核中的重复性工作：读取自研排版系统输出的 DOCX，执行规则化审核，输出报告，并对高置信格式问题自动修复。首版不追求全自动终稿，不用大模型直接裁决合规性。

## 2. 拆分智能体职责

排版审核不是单一动作，适合拆成多个专用智能体：

- 文档解析智能体：负责从 DOCX 提取结构和格式事实。
- 规则审核智能体：负责把事实与院校规范逐项比对。
- 安全修复智能体：负责执行允许自动修复的格式修正。
- 质量复检智能体：负责修复后二次审核，避免修复引入新问题。
- 报告智能体：负责把审核结果转换成 JSON、Excel、HTML。
- 迭代记忆智能体：负责记录高频问题、修复效果、人工复核压力和下一轮优化建议。
- 协调智能体：负责调度、上下文传递、失败兜底和批处理。

所有智能体共享 `SharedReviewState`，用来传递事实、产物、指标和决策。LLM 不是独立智能体，而是共享能力层。所有智能体后续都可以通过运行上下文访问同一个 LLM 服务，用于解释、归因、建议生成或规则配置辅助；硬规则裁决和自动修复仍由确定性逻辑控制。

LLM 配置契约固定包含：`provider`、`base_url`、`api_key`、`model`、`temperature`、`max_tokens`。

## 3. 定义共享上下文与契约

所有智能体通过统一的 `AgentRunContext` 和结构化结果通信。审核问题统一使用 `Issue` 数据结构，必须包含规则 ID、位置、实际值、期望值、严重等级、处理状态、置信度和建议。

## 4. 建立规则优先的执行策略

规则库是审核权威来源。智能体只执行规则，不把模型判断作为硬性合规依据。规则必须声明适用范围、期望值、严重等级、是否允许自动修复和置信度。

正式投产前必须先创建目标学校规则库。默认规则仅用于演示审核引擎，不代表任何学校官方要求。每个学校/学院/学历层次/模板版本都应维护独立规则文件，并记录官方规范来源、模板来源、发布日期和复核人。

## 5. 执行多智能体流水线

标准顺序为：

1. `DocumentParserAgent` 解析原文档。
2. `RuleAuditorAgent` 生成初审问题。
3. `SafeFixerAgent` 根据高置信问题生成修复文档。
4. `DocumentParserAgent` 解析修复文档。
5. `RuleAuditorAgent` 二次审核修复文档。
6. `QualityGateAgent` 比较前后结果。
7. 可选调用共享 LLM 能力层生成解释和人工复核建议。
8. `IterationMemoryAgent` 生成智能迭代画像，可写入长期记忆文件。
9. `ReportWriterAgent` 生成报告。

## 6. 测试与验收

用可控样本文档验证端到端链路：错误文档应能检出问题，启用安全修复后问题数量应下降，报告文件应完整生成。每新增一类规则，都要添加对应正例和反例。

## 7. 后续扩展

当基础审核稳定后，再接入渲染校验智能体处理目录页码、页眉页脚、跨页表格等视觉问题；共享 LLM 服务可替换为真实模型客户端，但必须保持“只解释、不裁决、不自动改文档”的边界，除非未来明确新增人工确认后的模型辅助修复流程。迭代记忆文件可用于统计排版系统高频漏改点、规则误报点和样本文档覆盖缺口。
